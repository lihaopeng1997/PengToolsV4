# PengToolsV4 需求管理右侧上下分栏修复需求 V1.0

> 本文为 Grok 开发文档。本轮只改 Private 版，标准安装包保持不变；用户本地数据必须兼容。已完成的夜间主题与响应式布局作为基线，不要回退。

## 一、问题与目标

### 问题描述

需求管理右侧区域分为上下两部分：
- **上区**：需求摘要（标题、事项类型、进度状态、目标系统、计划上线）+ 完成标记
- **下区**：文件库 / SQL / 发布与联动（Tab 切换）

用户期望：上下默认 3:7 比例（文件库占主导），允许拖动调整并持久化，需求摘要区内容紧凑不留大片空白。

当前存在三个问题：

1. **需求摘要区填充有问题**：内容集中在上部，下方留白；在大窗口下尤其明显，上区被截断后剩余空间变成空白条。
2. **上下拖动有问题**：用户拖动分隔条有上限卡住感，拖到一定程度就拖不动了；并且每次重新打开应用，用户上次拖动的比例会被强制重置，不尊重用户自定义。
3. **响应式模式切换时上下比例错乱**：窗口缩窄到 compact/narrow 模式时，上下分栏没有同步调整，可能出现上区过大挤掉文件库空间的情况。

### 目标

- 默认上下 3:7，文件库占大部分空间。
- 用户可自由拖动上下分隔条，拖动结果持久化，下次打开保持。
- 需求摘要区内容紧凑，不主动抢多余垂直空间，不留大片空白。
- 响应式模式切换时上下比例保持合理基线（窄屏下上区压缩，文件库仍占主导）。
- 旧数据兼容：缺失字段用默认值，不崩溃。

## 二、不改的范围

- **不创建第二套业务逻辑**：只修复上下分栏的尺寸分配和持久化逻辑，不改需求/文件库/SVN 的业务功能。
- **不改左右分栏**：左右 splitter 当前工作正常，不碰。
- **不改 Tab 结构**：文件库 / SQL / 发布与联动三个 Tab 保持不变。
- **不改完成标记的业务逻辑**：只改它父容器的尺寸表现。
- **不改配置文件字段结构**：`content_splitter_sizes` 字段名和格式保持不变，只是加载时不再强制纠正用户比例。
- **不改标准安装包**。

## 三、验收标准

### 3.1 默认比例

- [ ] 首次打开（无存储数据），上下区域按当前窗口高度 3:7 分配。
- [ ] 文件库区域明显大于需求摘要区域。
- [ ] 需求摘要区内容（标题、事项类型/进度状态/目标系统/计划上线、完成标记）紧凑展示，无大片底部空白。

### 3.2 拖动持久化

- [ ] 用户拖动上下分隔条后，松手后自动触发保存。
- [ ] 关闭重开应用后，上下比例保持用户上次拖动结果，不被强制重置。
- [ ] 用户把上区拖到 40% 甚至 50%，下次打开仍然保持，不回退到 30%。
- [ ] 拖动到极限位置（上区最小 / 下区最小）不崩溃、不折叠为 0。

### 3.3 响应式

- [ ] wide/standard 模式下保持 3:7 或用户自定义比例。
- [ ] compact/narrow 模式下上区压缩，文件库仍占大部分空间。
- [ ] 从 narrow 切回 wide 后比例恢复正常。

### 3.4 旧数据兼容

- [ ] 配置文件中无 `content_splitter_sizes` 字段时，使用 3:7 默认值。
- [ ] `content_splitter_sizes` 为非法值（0/负数/非数字）时，回退到默认值，不崩溃。
- [ ] 有合法 `content_splitter_sizes` 的旧数据，直接使用，不强制纠正比例。

### 3.5 主题与视觉

- [ ] 四套主题（含夜间安读）下，上下分隔条清晰可见、可拖动。
- [ ] 需求摘要区背景和边框在四套主题下正常。
- [ ] 完成标记按钮在摘要区中正常展示，不被挤压。
