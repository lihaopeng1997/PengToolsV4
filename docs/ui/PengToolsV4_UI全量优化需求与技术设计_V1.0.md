# PengToolsV4 UI 全量优化需求与技术设计

> 版本：V1.0  
> 日期：2026-07-20  
> 适用范围：PengToolsV4 Private（Windows / Python 3.12 / PyQt6）  
> 定位：供开发人员一次性完成 UI 成品化改造、供验收人员逐项核验的执行文档。

---

## 1. 背景、目标与边界

PengTools 是面向日常需求、SQL、SVN、文档及网关报文处理的本地效率工具。现有业务能力已经较完整，但界面仍存在“模块各自完成、整体未收口”的问题：部分页面像功能拼接，提示框和退出弹窗缺少产品感，需求管理左侧工具区布局不合理，加解密中的 XML 工具虽已可用但仍未形成完整工作台。

本轮目标不是换色或微调间距，而是建立一套可长期维护的企业级桌面端设计系统，并在不破坏现有业务逻辑的前提下完成核心页面重构。

最终体验必须同时满足：

- 企业级：信息层级清晰、克制可靠、批量操作安全、适合长时间办公。
- 年轻感：轻量科技感、适度色彩和图标反馈，不花哨、不游戏化。
- 模块有辨识度：需求管理、升级准备、加解密、设置各有工作场景特征，但仍属于同一产品。
- 懒人操作：默认推荐最常用路径，减少二次确认、重复填写和用户思考成本。
- 可维护：业务逻辑、组件逻辑和 QSS 主题职责清楚，后续新增功能不再复制粘贴样式。

### 1.1 不可突破的产品边界

1. 仅改 Private 私人版本；标准离线安装包 `PengToolsHub_Offline_Setup.zip` 不得包含或被私人功能影响。
2. 所有用户数据继续只使用 `config.local_data_dir()` 指向的 `data/`；升级仅替换程序文件，绝不覆盖、清空或迁移失败旧数据。
3. 自我学习模块仍由既有彩蛋入口保护；日报、需求管理、升级准备保持正常可见。
4. 内网 SVN 不在当前环境做真机承诺；只允许保留本地 SVN 工作副本的逻辑验证说明。
5. 本轮只做变更模块的定向测试，不要求全量回归。
6. 删除、覆盖、清空、提交等风险操作仍必须遵循：取消在左、确认操作在右、默认焦点为取消。

## 2. 现状代码基线与改造入口

本次设计必须在现有结构上成品化重构，避免无意义的大拆大改。以下文件为事实入口，修改前应使用 CodeGraph 再次确认调用关系。

| 场景 | 现有入口 | 本轮职责 |
|---|---|---|
| 主窗口、侧栏、关闭行为 | `main_window.py` / `MainWindow.closeEvent()` | 导航、窗口关闭决策、全局壳层 |
| 全局样式 | `resources/style.qss` | 色彩、控件、表格、树、对话框、Loading 的主题基线 |
| 控件尺寸与按钮语义 | `ui/design_system.py`、`ui/field_metrics.py` | 统一按钮、表面、字段尺寸；禁止页面自行造另一套规则 |
| 提示与确认 | `ui/confirm_dialog.py` | 信息、成功、警告、错误、危险确认、流程确认、退出决策 |
| 浮层 Loading | `ui/aurora_progress.py` | 非占位、非挤压布局的长任务进度反馈 |
| 网关加解密 / XML | `panels/gateway_panel.py`、`ui/xml_workspace.py`、`tools/xml_formatter.py` | 加解密工作台及 XML 全流程 |
| JSON 查看 | `ui/json_viewer.py`、`tools/json_viewer.py` | 深层数据可读性、搜索、复制和列宽 |
| 需求管理 | `panels/requirement_panel.py`、`tools/requirements.py`、`tools/svn_workspace.py` | 树结构、详情、文件、SQL、SVN 和台账联动 |
| 升级准备 | `panels/sql_panel.py`、`tools/release_prep.py` | SQL 整理、按系统生成、发版 Excel |
| 设置 | `panels/settings_panel.py`、`config.py` | 偏好、退出询问、长期可兼容的配置项 |

### 2.1 已确认的结构问题

1. `RequirementPanel` 的左树工具栏目前将“全选/删除”与“全部展开/全部折叠”拆成两行。该布局不是窗口宽度问题，而是代码结构问题，必须改为单行工具栏并重构左侧面板层级。
2. `ConfirmActionDialog`、普通提示框和关闭决策已开始统一，但视觉仍偏“定制 QMessageBox”，缺少明确的信息类型和产品化层级。
3. `GatewayDecodePanel`、`XmlWorkspace` 已有页面头、分栏与 XML 功能，但工具按钮散落，输入、输出、状态、错误定位之间的工作流仍需重组。
4. `JsonViewer` 已有可调整列宽的基础，仍需把深层 key、长 value、横向滚动、tooltip 与搜索定位作为完整可用性目标验收。
5. `AuroraProgress` 已是浮层实现，后续重点是统一触发范围、文案和所有成功/失败/异常收尾，不能为了“看起来高级”给普通刷新也加 Loading。

## 3. 总体设计方案

### 3.1 设计语言：Enterprise Pulse

整体采用“深色导航 + 柔和冷灰工作区 + 高识别操作色”的 Enterprise Pulse 语言。

- 基础：深靛蓝侧栏、雾灰蓝内容背景、白色/浅色卡片表面。
- 主色：靛蓝，用于主操作、当前导航、可点击核心元素。
- 成功色：绿色，只表示成功、已完成、已上线等正向状态。
- 警示色：琥珀橙，只表示需要关注、存在 SQL/通知/临时升级等业务标识。
- 危险色：低饱和红，只用于删除、不可逆覆盖、明确失败。
- 模块色：允许页面标题、状态条或图标点缀出现不同强调色，但不可重写全局按钮规则。

视觉 token 应收敛在 `resources/style.qss` 注释区与 `ui/design_system.py`，至少统一以下概念：`app-bg`、`surface`、`surface-muted`、`border`、`text-primary`、`text-secondary`、`primary`、`success`、`warning`、`danger`、`focus-ring`、`radius-control`、`radius-card`、`control-height`。

### 3.1.1 图标资产与按钮组合设计

本轮提供的首批离线图标位于 `resources/icons/`，采用统一 24px SVG 线性图标，线宽 1.8、圆角端点、`currentColor` 染色。它们不是装饰素材，而是界面语义的一部分：图标负责快速识别，文字负责避免歧义。

| 区域 | 图标 | 文字与按钮方案 |
|---|---|---|
| 左侧导航 | requirements / release / shield-key / settings | 20px 图标 + 模块名；激活时图标与文字同色、带窄色条 |
| 需求目录单行工具栏 | delete / expand / collapse | `全选` 用勾选框；`删除` 为带 delete 图标的危险幽灵按钮；展开/折叠为右侧紧凑图标文字按钮 |
| 文件与路径操作 | folder-open / copy / edit | 工具栏使用 18px 图标 + 短文字；表格行内只用图标按钮并提供 tooltip |
| SVN | lock / unlock | 锁状态显示在文件名之前；操作按钮图标 + “锁定/解锁”，不再使用文本状态列 |
| XML / JSON | xml / json / copy | 模式 Tab 使用图标 + 文字；格式化是唯一 Primary 按钮，粘贴/清空/复制为次级或幽灵按钮 |
| 反馈与弹窗 | success / warning / error / info | 24px 徽章，图标颜色只跟随消息语义；正文卡片不使用同色大面积填充 |

按钮组合必须遵守：关键且有歧义的操作使用“图标 + 文字”；仅在用户已熟悉的行内工具位置使用纯图标；所有纯图标按钮必须提供 tooltip 和 hover/focus 反馈；禁止使用 Emoji、彩色素材图标或网络图标库。

### 3.2 页面信息架构

所有业务页面统一采用四段结构；页面可根据内容省略某一段，但不可另造杂乱结构。

```text
页面标题区：图标 + 标题 + 一句可读的副标题 + 右侧状态/主动作
筛选或参数区：只放本页面开始工作所需的输入和筛选
主工作区：Tree / Table / Editor / Splitter / Card 等核心内容
反馈区：结果摘要、状态标签、非阻塞提示；不与主操作争抢空间
```

### 3.3 交互优先级

1. 用户先看到“当前要做什么”和默认推荐动作。
2. 常用动作放在可达位置；低频动作放入二级菜单或紧凑工具栏。
3. 一次操作只要求填写必要信息；能从需求、系统、当前选择中推断的字段自动带出，但必须可修改。
4. 成功只给轻量确认；失败给明确原因和下一步；风险操作才使用阻断式确认。
5. 所有可拖动区域必须具备可发现性：分隔条、表头边界、hover 指针或提示至少具备两项。

## 4. 详细功能与界面需求

## 4.1 主窗口、导航与模块区分

### 目标

让用户从任一模块返回时，都能明确知道自己位于同一软件内，同时识别当前模块的工作语境。

### 需求

1. 左侧导航保留深色基调，导航项使用统一图标、文字、激活背景和窄色条；禁止仅依靠文字加粗区分当前页。
2. 设置固定在侧栏左下区域，和业务导航区通过视觉分隔而不是另起标题栏。
3. 所有模块页面标题区统一为“图标 + 标题 + 副标题”；右侧可放模块级状态或主要动作，不放一排杂乱按钮。
4. 模块强调色建议：需求管理为蓝紫、升级准备为青蓝、加解密为靛蓝/紫、设置为中性灰蓝；强调色仅作用于标题图标、状态条、空状态插图等小面积区域。
5. 窗口最小尺寸继续保证核心操作不压缩到不可用；小窗口时优先隐藏次级文字或让工具栏横向滚动，不能截断主按钮文字。

### 验收

- 当前导航项在不阅读文字时也可识别。
- 设置不与顶部业务标题混在一起。
- 需求、升级准备、加解密、设置均有一致壳层且能一眼区分。

## 4.2 按钮、图标、输入与下拉控件

### 按钮语义

| 类型 | 适用场景 | 视觉规则 |
|---|---|---|
| 主按钮 Primary | 页面当前最重要的一个动作，如“生成发版文件”“开始解密”“格式化 XML” | 实色主色，单页面同一视区最多一个 |
| 次按钮 Secondary | 常用但非主路径，如“导入”“复制”“选择目录” | 浅表面、描边或弱填充 |
| 幽灵按钮 Ghost | 辅助、低风险、工具栏动作 | 透明/弱底色，必须有 hover |
| 危险按钮 Danger | 删除、清空、覆盖确认 | 红色语义，仅在风险上下文显示 |
| 图标按钮 | 展开、折叠、刷新、打开目录等高频小动作 | 图标必须可辨，tooltip 必须完整 |

### 控件统一要求

1. 按钮通过 `apply_button()` 或明确的统一 objectName 设置，禁止每个页面临时写一套高度、圆角、边框。
2. 所有图标优先使用本地 SVG/资源，保证离线打包；不得使用在线图标库、CDN、浏览器组件。
3. 下拉框必须有独立箭头区、可见边框、hover 和 focus 状态；日期控件必须能看出可输入也可弹出日历。
4. 输入框错误状态采用边框 + 简短字段级提示，不要仅弹全局错误框。
5. 禁用状态不可与普通状态混淆；禁用原因需要通过 tooltip 或字段说明可追溯。

### 文案规范

- 主动词优先：使用“导入需求”“生成发版清单”“锁定文件”，避免“确认”“提交一下”等模糊文字。
- 不用开发术语替代结果：用“正在生成发版清单”而不是“处理中”。
- 成功提示包含结果对象和下一步，不包含技术堆栈信息。
- 错误提示说明“哪里失败、为什么、可怎么处理”；原始异常只在可复制详情中展示。

## 4.3 全局提示框、确认框与退出决策

### 4.3.1 建立统一 Dialog API

在不破坏现有调用点的前提下，扩展 `ui/confirm_dialog.py` 的语义能力。页面只能调用统一 API，不再直接创建 `QMessageBox`。

| 类型 | 使用时机 | 图标/色彩 | 默认焦点 | 按钮规则 |
|---|---|---|---|---|
| 信息 Info | 无风险说明、空操作提醒 | 蓝灰信息图标 | 关闭/知道了 | 单按钮 |
| 成功 Success | 已保存、已导出、已完成 | 绿色对勾 | 关闭/知道了 | 单按钮，可选“打开所在位置” |
| 警告 Warning | 输入缺失、格式异常、可恢复问题 | 琥珀警示图标 | 关闭/返回修改 | 单按钮或安全取消 |
| 错误 Error | 失败、不可继续 | 红色错误图标 | 关闭 | 必须支持复制错误详情 |
| 危险确认 Danger | 删除、清空、不可逆覆盖 | 红色风险图标和风险说明卡片 | 取消 | 取消在左，危险动作在右 |
| 流程确认 Flow | SVN 更新、提交、批量生成等可影响外部状态的动作 | 蓝紫流程图标 | 取消 | 展示对象、范围、影响、可选备注 |

### 4.3.2 对话框结构

所有对话框使用如下固定结构，不允许只换颜色：

```text
图标徽章 + 标题 + 一句行动导向摘要
可选择/可复制的正文卡片（对象、影响、错误或结果）
可选设置区或详情折叠区
操作栏：安全操作在左，推进/危险操作在右
```

要求：

1. 每种类型至少在图标、强调色、标题语气、按钮、默认焦点中的三项以上存在真实差异。
2. 危险确认中必须显示风险对象，例如“将删除 3 条需求”，不要只有泛化的“确认删除”。
3. 对超长错误信息提供“复制详情”，默认正文保持可读，不撑爆窗口。
4. 删除/覆盖等确认按钮文本必须说出真实动作，例如“删除 3 条需求”“覆盖现有文件”。

### 4.3.3 退出弹窗重做

`ask_close_action()` 必须成为成熟的“关闭方式选择”弹窗，而不是普通确认框。

1. 标题建议为“关闭 PengTools？”；副标题直接解释：关闭窗口可隐藏至托盘继续运行，也可以彻底退出。
2. “隐藏到托盘”和“彻底退出”使用两个可点击选项卡，卡片内有图标、结果说明和当前推荐标记。
3. “记住本次选择，不再询问”放在独立偏好区；选中后立即明确其会影响下次关闭。
4. 默认动作由 `close_default_action` 控制，默认焦点仍落在安全路径“取消”。
5. 设置页面必须可重新开启“每次询问”，并可修改默认关闭动作；文案与弹窗保持完全一致。
6. 普通关闭调用 `hide()`；托盘退出调用完整 `_shutdown()`；两条路径绝不可合并。

### 验收

- Info、Success、Warning、Error、Danger、Flow 六类框一眼可区分。
- 危险确认按 Enter 不会误执行危险动作。
- 设置修改后，下一次关闭行为真实改变；恢复询问后再次弹窗。
- 退出到托盘不结束后台服务也不保留任务栏窗口；托盘“退出”才完全结束。

## 4.4 Loading 与异步任务规则

### 实现原则

继续使用 `AuroraProgress` 的浮层方向：不进入业务布局、不推动按钮、不改变右键菜单定位。需要统一它的状态、文案和任务生命周期。

### 必须显示 Loading 的操作

- SVN 拉取、更新、锁定、解锁、提交、检出。
- 扫描需求目录、导入文件、批量读取 Office 文档。
- 生成 SQL、写入或导出 Excel/Word/文本、创建发版文件。
- 任何预计超过约 500ms 的后台读取/写入任务。

### 不显示 Loading 的操作

- 普通刷新、筛选、标签切换、树展开/折叠。
- 选中行、打开详情、浏览文件列表、调整分隔条或列宽。
- 本地轻量复制、单个字段编辑、普通下拉选择。

### 统一生命周期

每一个长任务必须具备：启动前 `start_busy(语义文案)`；成功 `finish(完成结果)`；可预期失败 `fail(失败语义)`；异常捕获后先 `fail()` 再弹错误框；线程结束后恢复可操作状态。禁止仅在成功路径结束 Loading。

Loading 文案必须描述实际任务，例如“正在扫描 2026 年车险需求目录”“正在提交 4 个 SVN 文件”，禁止只显示“请稍候”。

## 4.5 加解密与 XML 工作台

### 目标

把 `GatewayDecodePanel` 从“解密页加 XML 按钮”升级为“离线报文处理工作台”，让用户顺着输入、处理、检查、复制的流程完成任务。

### 页面结构

```text
模块标题：网关加解密工作台 + 离线处理状态
工作模式：国密解密 | JSON 查看 | XML 格式化（可采用 Tab，但每个 Tab 均是完整工作区）
参数区：环境、密钥等必要参数，默认折叠或紧凑展示
主工作区：左输入 / 中央处理动作 / 右输出（或上下可拖动分栏）
结果区：格式状态、字符数、行数、错误定位、复制/导出等反馈
```

### XML 完整能力

`XmlWorkspace` 必须完整支持并保持离线：

1. 粘贴输入、清空输入、清空全部、复制输出。
2. 自动识别并处理外层英文双引号。
3. 自动处理常见转义文本，如 `\\n`、`\\t`、`\\\"`、Unicode 转义；不得错误破坏合法 XML 内容。
4. 一键格式化，保留 XML 声明，中文保持正常显示。
5. 输入输出分栏可拖动；大文本可换行或横向滚动，状态明确。
6. 格式错误必须提供“第几行、第几列、附近原因”；尝试将输入光标定位到错误行并突出显示。
7. 解密成功后，如果结果看起来是 XML，提供明显而不打断流程的“送入 XML 工具”动作；不得自动覆盖用户正在编辑的 XML。
8. JSON / XML / 原始文本的模式和结果边界清晰，不允许在同一文本框中混淆输入与输出。

### 操作区要求

- 主操作仅保留一个：当前模式下为“开始解密”或“格式化 XML”。
- 复制、清空、粘贴、换行等放入紧凑工具栏，并有图标、文字或 tooltip。
- 没有输入时主操作禁用并说明原因；格式失败不清空原输入。
- 密钥、明文、报文默认不落盘且不写日志，保持现有安全边界。

### 验收样例

- 普通 XML、外层双引号 XML、转义 XML、含 XML 声明的 XML、中文 XML 均能按预期处理。
- 非法 XML 给出行列定位且用户原文仍在。
- 长 XML 不会导致按钮或结果区不可达；分栏可拖动。

## 4.6 JSON 查看器优化

1. 深层节点的 key 列最小宽度必须保证可读；不可因缩进将字段名挤到不可辨认。
2. Tree header 的 key、类型、值预览列支持拖动；列边界视觉明确。
3. 长 key、长 value 提供横向滚动与 tooltip 双兜底；不能依赖固定超长列宽。
4. 搜索实时反馈匹配数，Enter/上下按钮跳转，下一个命中项自动滚入可视区并高亮。
5. 当前 JSONPath 明确可见并可一键复制；复制节点、复制路径与复制值的含义明确区分。
6. 不因页面样式改造破坏 `tools/json_viewer.py` 的解析与格式化逻辑。

## 4.7 需求管理全量界面重构

### 4.7.1 页面总体布局

```text
标题区：需求管理 + 当前系统/统计摘要 + 新增需求主操作
筛选区：全文搜索、系统、类型、状态（紧凑，且可清晰识别为下拉）
主体：左侧“需求目录” | 可拖动分隔条 | 右侧“详情与文件工作区”
```

左右分栏继续采用 `QSplitter`，最小宽度受控、不可完全折叠、尺寸持久化到既有 `requirement_ui.json`。拖动反馈必须可发现，拖动后不得丢失树节点、错误改变树层级或改变右键行为。

### 4.7.2 左侧需求目录

左侧不是“几组按钮 + 一个树”，必须重构为完整目录面板：

```text
目录标题区：需求目录 + 当前数量/筛选结果
单行工具栏：全选 | 删除 | 弹性间隔 | 展开 | 折叠
目录树：月份分组 > 需求/BUG 项 > 必要标记
轻量状态说明：SQL、通知周边、临时升级、锁定等标识说明（可选）
```

硬性要求：

1. “全选、删除、展开、折叠”必须在同一行；可以左右分组，但不得再拆成两行。
2. 顶层月份完整显示，使用较大的深色字重体现层级；不要隐藏顶层日期标题。
3. 子项仅将需求号/BUG 号设为高亮；需求标题使用正常字重，标记使用更浅或语义色，避免所有文字都加粗造成混乱。
4. 树项目标题来源为 SVN 路径截取部分或用户可修改的目录标题，不再重复显示无意义的“需求号 + 标题”双重内容。
5. SQL、通知周边、接口文档、临时升级、锁定等用小型语义徽标或图标表达；不可把整行涂成高饱和色。
6. 支持多选、全选、Delete/Backspace 批量删除、右键编辑、右键打开文件夹、拖动至其他月份；拖动时应只更新目标月份，不允许节点消失。
7. 选择、hover、展开、禁用/空状态均需有明确但克制的视觉反馈。
8. 目录筛选后的全选只针对当前可见且可删除的需求/BUG；删除框必须显示实际数量和标题摘要。

### 4.7.3 右侧详情与文件工作区

1. 移除无价值的“需求附件框”和只读说明框；右侧以完整目录内容为主体。
2. 顶部保留可编辑的核心信息：系统、标题、类型、状态、上线月份、计划上线、实际上线、开发分支 SVN、需求/BUG 路径等。日期字段可手输也可弹出日期选择。
3. SVN 路径与本地路径完整保留，提供复制和打开文件夹；路径过长时可省略显示，但 tooltip/复制必须得到原始全路径。
4. 文件表完整展示目录文件：名称、修改时间、类型、大小。表头列必须可拖动、边界清晰、支持合理的列宽记忆。
5. 文件夹可展开，双击/右键可打开文件或文件夹；已锁定文件在文件名前显示锁图标，不再单独放冗余 SVN 状态列。
6. 文件树与关联 SQL 区继续支持上下分割并持久化尺寸；在窄窗口也不得压扁主要列。
7. “关联 SQL”应可见当前选择需求的 SQL 汇总、来源和状态；从需求管理、SQL 整理、升级准备之间的信号继续互通。

### 4.7.4 搜索、分类与日期

1. 全文搜索必须实时更新目录和右侧定位；搜索范围包括编号、标题、目录标题、说明、SQL、系统、路径和关键标记。
2. 需求目录按上线月份倒序分组；同月按修改日期倒序；无日期数据归入明确的“未分类”分组而非静默丢失。
3. 每个分类/月份标题需显示数量，筛选后反映当前数量。
4. 不再保留没有实际意义的“全部分类”下拉框；只保留系统、类型、状态等真正可帮助筛选的字段。

### 4.7.5 SVN 与保存规则

1. 新增需求、编辑需求、粘贴 BUG 时开发分支 SVN 地址允许为空；保存不得因此被拦截。
2. 存在本地 SVN 工作副本时，自动回填 SVN 地址；不在本地验证内网地址。
3. 锁定/解锁仅对真实选中文件可用；文件名前锁图标是状态展示唯一来源。
4. 拉取、更新、锁定、解锁、提交使用流程确认和 Loading；刷新文件树不使用 Loading。
5. 所有文件夹场景应统一提供“打开文件夹”，包括左树右键、右侧路径区、右侧文件树右键。

### 验收

- 左侧四个工具在同一行，布局在 960px 宽窗口仍可操作。
- 顶层月份、子项标题、编号、业务徽标层级清晰，未发生拖动后消失。
- 目录、文件表、关联 SQL 区可拖动、可持久化、可打开文件夹。
- 搜索、日期排序、批量删除与 SVN 分支为空保存均可定向验证。

## 4.8 升级准备与需求联动

本轮以体验收口为主，不能破坏既有业务链：需求/BUG -> SQL -> 升级准备 -> 发版 Excel。

1. 选择升级日期后才进入生成主流程，但已填写草稿不得因切换日期丢失。
2. 系统是一级上下文：选择需求、生成 SQL、生成发版 Excel 均按系统隔离；一个人多个系统开发时可分别生成。
3. 优先推荐计划上线/实际上线日期匹配的需求和 BUG，用户可勾选、取消或搜索。
4. 任务编码从需求路径自动提取但允许修改；开发分支只取 SVN 最后一段目录名，分支为空允许保留空白。
5. 发版 Excel 中当天 sheet 不存在则复制模板样式创建，存在则在尾部追加；必须保持标题、列宽、行高与模板一致。
6. 生成 SQL、导出 Excel、SVN 提交等长任务有 Loading；普通选择需求和调整字段不显示 Loading。

## 4.9 设置页

1. 设置页面保持可滚动，新增内容不得遮挡现有说明或关键设置。
2. 增加清晰的“关闭行为”分组：每次关闭时询问、默认隐藏到托盘、默认彻底退出、恢复默认。说明文字必须使用产品文案。
3. “不再提示”不是孤立开关：关闭弹窗勾选后同步到设置；设置调整后立刻影响下一次关闭。
4. 清理或重置偏好属于风险操作，必须使用危险确认，不得误清 `data/` 业务数据。
5. 所有设置 JSON 新字段以默认值和 `setdefault` 方式兼容旧版本；写回时保留未知旧字段。

## 5. 技术实施方案

## 5.1 分层原则

```text
resources/style.qss          主题 token、控件外观、统一 objectName 样式
ui/design_system.py          组件语义：按钮、表面、状态、图标辅助
ui/confirm_dialog.py         对话框 API、类型与默认焦点规则
ui/aurora_progress.py        全局浮层 Loading 生命周期和视觉
panels/*.py                  页面结构、业务绑定、仅调用统一组件
tools/*.py                   纯业务与解析逻辑，不因 UI 调整掺入 QWidget 依赖
config.py                    数据目录、设置默认值、旧数据兼容
tests/                       本轮受影响模块的定向行为测试
```

禁止将大量 QSS 字符串散落在面板中；禁止为新页面复制另一份提示框、Loading 或按钮实现；禁止在工具层直接 import UI。

## 5.2 推荐实施顺序

1. **基线与设计系统**：审计 `style.qss`、`design_system.py`、`field_metrics.py`，整理 token、按钮、表面、输入、表格、树、图标规范。
2. **全局反馈组件**：完成 `confirm_dialog.py` 与 `aurora_progress.py`，先让所有页面可复用统一 API。
3. **窗口与设置联动**：完成关闭决策、托盘路径、设置项兼容与文案收口。
4. **加解密工作台**：重构 `gateway_panel.py` 与 `xml_workspace.py` 的布局和 XML 交互，保持 `tools/xml_formatter.py` 纯逻辑。
5. **需求管理**：先改左侧目录结构和树交互，再整理右侧详情/文件分区，最后验证 SVN 和 SQL 联动。
6. **JSON 与全局同步**：完成 JSON 可读性及 SQL/升级准备/设置的视觉收口。
7. **定向验证与打包**：仅验证变更模块，构建 Private 包，检查 `data/` 未被纳入包体，提交并推送 GitHub。

## 5.3 数据兼容方案

1. 不新增数据库；继续采用本地 JSON 和既有文件目录。
2. 新 UI 偏好如 splitter 大小、列宽、关闭行为写入既有对应 JSON；读时提供默认值，写时保留历史未知字段。
3. 用户数据不可因页面重构重新分类、静默覆盖或删除。涉及批量迁移时必须预览和确认；本轮原则上不做数据结构迁移。
4. 打包脚本不得携带开发态 `data/`，升级脚本不得删除安装目录 `data/`。

## 5.4 性能与稳定性

1. Office 扫描、目录扫描、SVN 任务继续后台执行，UI 线程只更新进度和结果。
2. 搜索使用轻量即时过滤；文件内容全文解析应防抖或在数据加载后缓存，避免每次输入重新读磁盘。
3. Tree 拖动使用当前已采用的延迟更新原则，先让 Qt 完成事件，再更新数据模型，避免节点丢失。
4. 表格和树的列宽、分隔条位置应在用户操作完成后再持久化，避免拖动时频繁写文件。
5. 所有线程/回调在窗口关闭和面板销毁时安全断开，避免退出后回调访问已释放控件。

## 5.5 离线与安全

- 不引入 HTTP、WebSocket、在线 CDN、浏览器内核或云同步。
- 图标、字体依赖使用仓库本地资源并进入 Private 构建。
- 加解密输入、密钥和解密明文不默认落盘、不写日志。
- 错误复制允许复制必要诊断信息，但不得在默认 UI 中暴露密钥或完整敏感报文。

## 6. 测试与验收方案

只做本轮影响模块的定向验证。每一项应至少有函数级测试或可复现的人工操作记录；不以“代码看起来没问题”替代验证。

| 专项 | 最小验证 |
|---|---|
| 关闭与设置 | 询问关闭、记住选择、恢复询问、隐藏到托盘、托盘彻底退出 |
| Dialog | 六类提示视觉/按钮语义；删除确认默认焦点为取消；错误可复制详情 |
| Loading | 导入/导出/SVN 长任务有浮层且异常消失；刷新/筛选/展开无 Loading、无布局跳动 |
| XML | 正常、外层引号、转义、中文、声明、错误行列定位、复制、清空、分栏拖动 |
| JSON | 深层 key、列拖动、长值 tooltip/横滚、实时搜索与自动定位 |
| 需求目录 | 工具单行、月份层级、多选删除、右键、拖动不丢节点、筛选排序、系统筛选 |
| 文件工作区 | 全列展示、列拖动、文件夹打开、锁图标、路径复制、分区尺寸持久化 |
| 需求/SVN | 分支为空保存；本地 SVN 回填；锁/解锁/更新/提交的确认与 Loading |
| 升级准备 | 多系统隔离、需求联动、日期推荐、发版 Excel sheet 新建/追加与格式一致 |
| 升级兼容 | 使用已有 `data/` 启动，设置和 splitter 数据仍可读取；构建包不含用户 data |

### 验收分级

**通过**：本文件的核心专项均已落地，页面结构有真实重构，视觉和交互具备统一产品感，定向测试通过，Private 包可运行。

**部分通过**：主要功能可用但存在明显拼接感、关键页面未按结构重构、或某个核心专项缺少验证。

**不通过**：仅改颜色或样式；四个树工具仍为两行；提示框只换文案；XML 仍是附属按钮集合；风险操作默认可误触。
