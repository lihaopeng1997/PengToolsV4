# PengToolsV4 接口排查与启动体验问题优化需求 V1.0

## 0. 本轮范围

本文件只作为 Grok 开发需求，Codex 本轮不直接修复代码。

需要解决三个问题：

1. 黑色内网任务栏上，软件窗口/任务栏图标辨识度不足；同时软件允许多开，第二次双击没有“软件已打开”的反馈。
2. 接口排查显示“监听成功”，但请求列表和接口详情没有内容；接口排查应升级为接近 Fiddler2 的本机抓包工作台，不再强制用户先选择浏览器。
3. 加解密模块的解密参数输入区域被隐藏或不明显，用户找不到录入 Key、IV、算法参数的位置；必须恢复清晰可见的参数区。

标准安装包必须保持不变，只构建 Private 版。用户数据、历史配置和本地文件必须兼容保留。

---

## 1. 问题一：任务栏图标与单实例

### 1.1 高对比任务栏图标

目标是在黑色 Windows 任务栏和浅色任务栏上都清晰可见。

- 运行时窗口图标、任务栏图标、托盘图标、悬浮窗图标分别验证，不要只替换侧栏 Logo。
- 新增专用高对比任务栏图标：浅色/白色圆角底，深墨绿或深灰标志，保留 PengTools 品牌形态。
- 图标边缘必须有明确轮廓，不能使用接近纯黑的背景，也不能使用低对比度灰绿图形。
- ICO 必须包含 16、20、24、32、48、64、128、256 多尺寸；SVG 可作为运行时高 DPI 来源，但 EXE/任务栏必须有可靠 ICO 回退。
- 图标要在黑色任务栏、浅色任务栏、任务栏小图标模式和 Alt+Tab 中分别检查。
- 主题切换不得改变任务栏图标可读性；任务栏图标不能跟随夜间主题变成黑色不可见。
- 不得影响窗口内部按钮、导航图标和悬浮窗的现有主题 tint。

### 1.2 单实例设计

使用 Qt 本机 IPC（推荐 `QLocalServer` + `QLocalSocket`，或等价的 Windows 命名互斥/本地 IPC），禁止用固定临时文件锁代替。

启动流程：

1. 首个进程启动时创建唯一服务名，例如 `PengToolsHub.Private.4.27`，并保存服务对象生命周期。
2. 第二次启动先连接该服务；连接成功后发送 `activate` 消息，然后立即退出第二进程。
3. 首进程收到消息后：
   - 如果最小化，恢复窗口；
   - 如果隐藏到托盘，显示主窗口；
   - `raise_()` + `activateWindow()` 抢回焦点；
   - 弹出非阻塞或可关闭提示：“PengTools 已打开，已为你切换到正在运行的窗口。”
4. 第二次双击不能创建第二个托盘、定时器、后台监听器、SVN worker 或浏览器调试会话。
5. 首进程异常退出后，第二次启动不能被僵尸锁永久阻塞；服务名不可用时要能安全清理并重新创建。
6. Private 与标准版使用不同服务名，不能互相误判为同一个软件。
7. 关闭/退出流程要在应用退出后释放本地服务；普通最小化到托盘不释放服务。

验收：连续双击 EXE 10 次，任务管理器中只能有一个主进程/一套托盘；每次第二次启动都能唤醒首个窗口并显示提示。启动参数、数据目录、版本升级后服务名要保持兼容。

## 2. 问题二：接口排查升级为 Fiddler 式本机工作台

### 2.1 产品方向

用户不应先理解 CDP、调试端口或浏览器 EXE 才能抓接口。默认体验应接近 Fiddler2：

```text
启动 PengTools → 点击“开始监听” → 正常操作浏览器/IE → 请求自动出现在列表
```

浏览器选择和调试端口保留为高级能力，而不是首要入口。

### 2.2 监听模式

提供清晰的监听模式卡片：

#### 模式 A：本机通用代理（默认）

- 使用 `mitmproxy` 或等价本地 HTTP/HTTPS MITM 技术监听 `127.0.0.1`。
- 适用于 Chromium、IE 以及遵循 Windows/WinINet 代理的其他浏览器和桌面程序。
- 启动前明确显示将临时修改当前用户代理；停止监听、软件退出、异常恢复时还原原代理。
- HTTPS 首次使用需要用户明确确认安装本机用户证书；证书只允许用于本机调试，提供移除入口。
- 不得静默修改系统代理，不得修改其他用户代理，不得监听外网端口。
- 代理启动失败时必须显示真实原因（端口占用、证书权限、WinDivert/mitmproxy 依赖等），不能仍显示“监听成功”。

#### 模式 B：Chromium CDP（高级）

- 保留自动识别、手选浏览器 EXE、启动独立调试浏览器、连接已有 `127.0.0.1` 调试端口。
- 不打包任何浏览器 EXE，不增加安装包体积。
- CDP 仅作为无法使用系统代理时的精确模式，不要求所有用户先配置浏览器。
- Firefox 明确提示首版不支持实时监听。

#### 模式 C：IE 代理（兼容入口）

- 复用通用代理的 WinINet 备份、恢复和证书流程；界面上说明 IE 属于兼容模式。
- 不再让用户在 IE 模式下寻找 Chromium 浏览器选择框。

### 2.3 “监听成功但没有请求”的修复要求

这是 P0 级 BUG，必须从事件链路和状态机同时修复。

- “开始监听”按钮只有在以下条件都成立后才显示成功：监听 worker 已启动、代理/CDP 通道已建立、端口可连接、事件回调已注册。
- 建立监听后显示实时状态：监听模式、绑定地址、端口、代理是否生效、已捕获请求数、最近一条请求时间。
- 如果 10 秒内没有请求，不要直接判定失败；显示“监听已建立，等待浏览器请求”，并提供“检查代理/重新连接”入口。
- 如果事件通道断开，状态改为“已断开”，停止显示“监听中”。
- 请求抓到后必须先进入内存会话，再通过 Qt 主线程 signal/slot 或安全队列更新表格；禁止从 worker 线程直接操作 QWidget。
- CDP 事件必须订阅并测试：`Network.enable`、`Network.requestWillBeSent`、`Network.responseReceived`、`Network.loadingFinished`、`Network.loadingFailed`。
- 事件合并必须使用稳定的 `requestId`，不能因 response 先于 request、重定向或 loadingFailed 而丢弃记录。
- 代理 flow 必须在 request/response 两阶段均能生成记录；没有响应体时仍展示 URL、方法、状态、请求头和失败原因。
- 不得让默认筛选器误隐藏所有业务接口；默认显示 XHR/Fetch、Document 和未知类型，静态资源可默认隐藏但必须有明确“显示静态资源”开关。
- 抓到第一条记录后自动刷新计数与列表；新请求到达不能覆盖当前已选详情。
- WebSocket、SSE、跨域、重定向、HTTP 失败、空响应体要以状态说明展示，不得静默消失。
- 对响应体 `getResponseBody` 失败时保留请求摘要，不得因此删除整条接口。
- 内网设置了 HTTP_PROXY/HTTPS_PROXY 时，CDP 的 `127.0.0.1` WebSocket 和 HTTP 调试请求必须绕过代理。

### 2.4 Fiddler 式界面

左侧会话列表：

- 默认列：状态、方法、host/path、耗时、类型、时间。
- 可选列：大小、来源、失败原因。
- 实时搜索 URL、host、path、method、状态码、Content-Type。
- 筛选：全部、XHR/Fetch、失败、慢请求、JSON/XML、静态资源。
- 显示 `当前显示数 / 总捕获数`。
- 行颜色只做轻量状态提示，不使用刺眼整行红底。

右侧详情：

- 概览：URL、方法、状态、耗时、类型、来源、大小、失败原因。
- 请求：Query、Headers、Cookie、Body。
- 响应：Headers、Body。
- 草稿：本地地址、重写 URL、Postman v2.1、cURL。
- JSON/XML 自动格式化；可送入格式工具、加解密模块。
- Cookie、Authorization、Token 默认脱敏；复制/显示敏感内容需提示。
- 明确展示“仅生成草稿，不发送请求”。

### 2.5 安全边界

- 报文、Cookie、Token、密钥、证书私钥不得写入磁盘、日志、handoff 或崩溃信息。
- 仅保存浏览器路径、端口、本地地址、列宽和筛选偏好。
- 停止监听、关闭页面、退出应用后清空捕获内存。
- 只绑定 `127.0.0.1`，拒绝 `0.0.0.0`、局域网和公网监听。
- 不增加任何云端、外网或业务接口发送能力。

## 3. 问题三：加解密参数必须清晰可见

### 3.1 交互要求

在加解密模块中，解密参数区必须固定显示，不得默认折叠或藏在“更多”里。用户打开解密功能后，应立即看到：

- 算法/模式：AES、DES、RSA、SM4 等现有支持项
- 编码：UTF-8、Base64、Hex 等
- Key 输入框
- IV/Nonce 输入框（算法需要时显示）
- 密钥格式/填充方式/字符集等已有参数
- 解密按钮、清空按钮、复制结果按钮

### 3.2 动态参数规则

- 根据算法显示或隐藏 IV、Padding、Mode 等参数，但隐藏时必须在界面说明“当前算法不需要此参数”。
- Key 不得使用过短的单行高度或仅显示密码圆点；默认显示可录入位置，提供“显示/隐藏 Key”切换。
- 参数标签必须写清楚“Key（密钥）”“IV/Nonce（初始向量）”，不能只写 Key、Param 等缩写。
- 输入框 placeholder 给出示例格式，但不得把真实密钥写入示例或日志。
- 解密失败时保留 Key/IV 输入内容，提示具体原因，不得清空用户刚录入的参数。
- 已有 JSON 解密行为、数据结构和敏感信息不落盘规则不能破坏。
- 从接口排查中心送入加密报文时，只自动填充报文，不覆盖用户已录入的 Key/IV；若为空可提示用户录入。
- 加解密模块的响应式布局：宽屏参数区与报文区并列，窄屏参数区置于报文上方；Key 输入框在最小宽度下仍可见。

## 4. 技术改造建议

- `run.py`：单实例服务、窗口激活、图标资源加载。
- `resources/brand/`、Private spec：高对比 ICO/SVG 和打包资源。
- `tools/browser_debug.py`：CDP 本机连接、代理绕过、事件订阅、事件合并和断线状态。
- `tools/ie_proxy.py`：代理状态机、证书和 WinINet 恢复。
- 新增或整理 `tools/interface_capture.py`：统一代理/CDP/IE 记录模型，Panel 不直接处理底层 flow。
- `panels/interface_debug_panel.py`：Fiddler 式三栏工作台、状态胶囊、空状态、事件计数和详情联动。
- 加解密面板：参数区视图模型与算法参数显隐，不改变既有加解密核心逻辑。
- 所有配置读写使用 `setdefault` 兼容旧数据；捕获报文不得进入 `data/interface_debug.json`。

## 5. 定向验收

### 单实例与图标

- 黑色任务栏、浅色任务栏、Alt+Tab、托盘、悬浮窗分别检查。
- 连续双击 10 次，只保留一个主进程和一套托盘。
- 第二次启动能唤醒首窗口并提示“软件已打开”。
- 首进程异常退出后可重新启动；Private 与标准版互不冲突。

### 接口监听

- 不选择浏览器，启动通用代理后在 Chrome、Edge、IE 中访问内网页面，均能看到请求。
- Chromium CDP 手动连接仍可用。
- 验证请求、响应、XHR/Fetch、重定向、失败请求、慢请求、JSON、XML、空响应和静态资源筛选。
- 验证代理停止、异常退出、证书安装/移除和原代理恢复。
- 验证 10 秒无请求时状态准确，不误报“监听成功但无数据”。
- 验证停止监听和退出后内存会话清空，配置文件不含报文和密钥。

### 加解密

- 打开解密功能即可看到 Key 和必要参数。
- 不同算法切换时参数显隐正确。
- Key/IV 不被错误清空、不落盘、不写日志。
- 从接口排查送入报文后，参数区仍然可编辑。

只运行本轮相关定向测试，构建并启动 Private 版；标准安装包不得修改。完成后在 `.codex_work/grok_handoff/` 写摘要，提交并推送 GitHub main。
